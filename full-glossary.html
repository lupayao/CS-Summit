Git & GitHub Terms Explained

Repository (Repo)

A folder where all your project files and their history are stored. Think of it as a project binder that keeps track of every change you make.

Commit

Saving a snapshot of your work. Each commit records what changed and lets you go back to that point if needed.

Branch

A separate line of work in your project. You can try new ideas on a branch without changing the main project until you’re ready.

Merge

Combining changes from one branch into another. This is how you bring everyone’s work together.

Pull Request

Asking to add your changes to the main project. Others can review your work before it’s included.

Clone

Making a copy of a project from GitHub onto your own computer so you can work on it.

Push

Sending your saved changes (commits) from your computer up to GitHub so others can see them.

Pull

Getting the latest changes from GitHub onto your computer, so your copy is up to date.

Staging Area (Index)

A place where you prepare files before saving them in a commit. Like a waiting room for your changes.

Remote

A version of your project that lives on the internet (like on GitHub), so you and your classmates can share work.

Fork

Making your own copy of someone else’s project on GitHub, so you can experiment without affecting the original.

Conflict

When two people change the same part of a file, Git can’t decide which change to keep. You have to choose how to fix it.

Stash

Temporarily putting aside changes you’re not ready to save, so you can work on something else.

Tag

A label for a specific point in your project’s history, often used to mark releases or important versions.

Revert

Undoing a specific commit, putting your project back to how it was before that change.

Rebase

Moving or combining a series of commits to make your project history cleaner. Advanced, but useful for organizing work.

Blame

Shows who last changed each line in a file. Helpful for tracking down when and why something changed.

History

A list of all the commits made in your project, so you can see what changed and when.

README

A file that explains what your project is, how to use it, and any other important info. Usually the first thing people see on GitHub.

Issue

A way to track tasks, bugs, or questions about a project. Like a digital to-do list or help desk.

Action

Automated tasks that run on GitHub, like checking your code or deploying your project.

License

A file that tells others how they can use, share, or change your project.

Sync

Making sure your local project and the remote project on GitHub have the same changes.

Cherry-pick

Taking a specific commit from one branch and applying it to another branch.

Squash

Combining several commits into one. This helps keep your project history tidy.
